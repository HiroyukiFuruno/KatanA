# KatanA Workflow & Skill Orchestration Guide

このスキルは、KatanAリポジトリに多数存在する **個別のWorkflow（`/`コマンド）やSkill** を、いつ・どの順番で・どのように組み合わせるべきかをAIエージェントに指示する「羅針盤（Compass）」として機能します。

AIエージェントは、ユーザーから「XXXをしたい」「リリースしたい」「一括変更したい」とアバウトな指示を受けた際に、必ずこのガイドを参照して **最適なスキルやワークフローへの導線** を提示し、確実で安全なプロセスを遂行してください。

## 1. 新規機能・バグ修正の標準プロセス (OpenSpec + Task Micro Cycle)

タスクの粒度が大きい（または仕様から整理する必要がある）場合、AIは以下の順序でプロセスを提案・実行します。

1. **仕様定義 & 起票**: `/opsx-propose` または `/opsx-new`
   - これにより `openspec/changes/...` 配下に必要なアーティファクトが生成されます。
2. **実装フェーズ (One-by-one)**: `/task-micro-cycle` または `/opsx-apply`
   - KatanAは「一気に全実装」を嫌います。`tasks.md` に定義された個別のタスク（例: `1.1`, `1.2`）単位で `/task-micro-cycle`（【実装→自己レビュー→承認待ち→コミット】の厳格なループ）を回します。
3. **コミット & プッシュ**: スキル `commit_and_push`
   - マイクロサイクル完了後、Pre-commit の検証をパスさせ、必ず **日本語** でコミットメッセージを作成します。
4. **マージ準備**: スキル `create_pull_request`
   - PRテンプレートに従い、適切に GitHub PR を作成します。
5. **クローズ**: `/opsx-archive`
   - 変更が完了・マージされたら、OpenSpecのフォルダをアーカイブします。

## 2. 大規模なリファクタリング・複数ファイルの変更

**危険を伴う一括変更**（例：共通コンポーネントのリネーム、複数ファイルにまたがる仕様変更）では、速度よりも安全性を優先してください。

- **必須スキル**: `bulk_modification_protocol` (一括＝早いは幻想 をテーマにした厳格なワークフロー)
- **行動指針**: 具体的な置換を実行する**前**に、必ず `commit_and_push` による現在のSavepoint（セーブポイント）の作成を促します。そして、変更計画を詳細にユーザーに提示・同意をとってから `multi_replace_file_content` を駆使して変更を加えます。

## 3. アプリケーションのリリースプロセス (vX.Y.Z)

ユーザーから「KatanAのリリースをお願いします」と指示された場合は、他のプロセスを停止し、以下のリリース用スキルを単独で起動してください。

- **必須スキル**: `release_workflow`
- **内容**: `make release` によるバージョン不整合チェックの修正、CHANGELOG更新、および GitHub Actions による自動リリースフロー（ビルド・アテステーション・アセットアップロード）を一気通貫で定義しています。AIはこれに従い、一歩ずつユーザーと確認しながら進行します。
- **補足スキル**: リリースに付随する `CHANGELOG.md` の更新は `changelog-writing` スキルを併用して、英語（UTC）と日本語（JST）を正しく同期させます。

## 4. 日常的な品質保証・コードレビュー

実装途中や、コミット実施の直前にユーザーから「レビューして」「Lintエラーが出た」と言われた際に挟むプロセスです。

- **Pre-commit Self-Review**: スキル `self-review`
  - コーディング規約（Rust/Svelteなど）、破壊的変更の有無を自律的にチェックするスキル。
- **UI/UXに関するレビュー**: スキル `ui-ux-pro-max` (Webデザイン向けだが、Katana UIにおけるEguiの一般的な配置などでヒントにする)

### エージェントへの絶対命令 (Agent Directive)

1. ユーザーから漠然とした「作って」という依頼が来た場合、「仕様から練る `/opsx-propose` を起動しますか？」または「一括変更のための `bulk_modification_protocol` を適用しますか？」と **具体的なワークフロー/スキル名** を提示して選択させてください。
2. 決して、ユーザーに言われるがまま「全ファイルを一気に書き換えて放置する」という行動をとってはいけません。
3. KatanAでは **「検証（Verification）」** と **「独立したレビュー（Self-Review）」** によってのみコードが正となります。

## 5. ブランチのクリーンアップ (Branch Pruning)

**重要:** リリース完了時や OpenSpec 完了・統合後は、不要になったローカル・リモートブランチがゴミとして溜まらないよう、**必ず** クリーンアップを実施してください。
AIエージェントは変更完了後に `git fetch --prune` および `git branch -v` / `git branch -r` を使って、不要な `feature/` や `hotfix/` ブランチが残存していないかを点検・削除（ユーザーへの提案）する義務があります。
