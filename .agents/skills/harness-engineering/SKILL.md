# Harness Engineering Protocol (Agent Self-Improvement)

## 1. 目的と定義 (Philosophy)

本スキルは、AIエージェントである「私自身」が、システムの自律的な開発・保守をスケールさせるために**「自分自身が確実に動作するための環境（Harness）」を継続的に設計・自己改善していくためのワークフロー**です。
「エージェントがミスをした場合、二度と同じミスを繰り返さないように解決策（仕組み）をエンジニアリングする」という Mitchell Hashimoto 氏や OpenAI チームが提唱した概念に基づきます。

## 2. ハーネスエンジニアリングの3本柱

エージェントとして開発に参加する際、単に「コーディングタスク」をこなすのではなく、以下の3つの役割（Harness設計）へシフトしなければなりません。

1. **System (システムの構築):**
   * CI/CD、マージゲート、解析基盤などの「開発全体が安定して回る仕組み」を自ら提案・整備する。
2. **Scaffolding (AIのための足場づくり):**
   * リポジトリ構造の最適化、専用ツールの作成、自動化スクリプトの整備により、将来の自分（AI）が動きやすい構造を作る。
   * プロンプトだけで制御困難な「暗黙の文脈」を排除し、ツールのスクリプト化やテストの形で明文化する。
3. **Leverage (レバレッジの最適化):**
   * 一時しのぎの回避策（AI Slop）を蓄積しない。部品化と抽象化を行い、「深さ優先（Depth-first）」で再利用性の高い仕組みを育てていく。

## 3. 実践のルール (Core Rules)

AIエージェントがタスクを実施中、または自己のミス（失敗）に気づいた際は、直ちに以下の原則に沿って環境の適応（自己改善）を実施してください。

### A. 機械的制約への昇格 (Enforcing Constraints)

「AI自身が自然言語の指示で気をつける」というアプローチはすぐに崩壊します。

* エラーを修正する際は、単にバグを直すだけでなく、**カスタムLinterの追加、Pre-commitフックの強化、アーキテクチャテストの拡充**によって、システム的に再発を防止してください。
* エラーメッセージは「AI自身が読んで次にどう直せばいいか自明なメッセージ（エージェントフレンドリーな出力）」にカスタムしてください。

### B. 知識基盤の構造化 (System of Record)

肥大化した１つのドキュメント（巨大なAGENTS.md等）はAIのコンテキストを圧迫し幻覚を招きます。

* 新たな暗黙知や設計上の決定事項（Core beliefs）を発見した場合は、`.agents/skills/` や `docs/` 配下に小さく分割して記録（Codify）し、エージェントが「必要な時に必要な知識だけ（Legibility）」を取得・把握できるように整理してください。

### C. 把握可能性の向上 (Legibility)

SUT（テスト対象）がAIから見て「把握不可能＝Legibilityが低い」場合、実装に着手してはなりません。

* UIを直す前にDOMスナップショット機能やスクショツールをHarnessに組み込む。
* 使い捨てスクリプト(`scratch.rs`等)での検証を絶対禁止し、公式のテストランナーから呼び出せるドライバ環境やモックを先に作る。
* 「自分（AI）が見える、計測できる」状態になるまでHarnessへの投資を優先してください。

## 4. トリガー基準

* 🚫 何度も同じような修正ミスやLinter違反を繰り返していると気付いたとき。
* ✅ バグ修正完了後、「このバグはどうすればCIで自動検出できたか？」を考えたとき。
* ✅ 人間から「この変更プロセスを自動化・永続化しておいて」と指示を受けたとき。
* ✅ 未テストの巨大なロジックに手を入れるよう求められ、自分の出力に確信が持てないとき（まずHarnessを組む）。
