# 改善提案: コード品質と安定性の向上 (リファクタリング / 不具合の可能性)

## リファクタリング: `unwrap()` および `expect()` の多用
現在のコードベースでは、`crates/katana-ui/src/main.rs` や `http_cache_loader/tests.rs` などを中心に `unwrap()` と `expect()` が1000箇所以上で使用されています。
テストコード内での使用は問題ありませんが、アプリケーションのメインロジック（特に起動時のフォント読み込みや設定ファイルのパースなど）での使用は、予期せぬパニックによるクラッシュを引き起こす原因となります。
`Result` と `?` 演算子を使用した適切なエラーハンドリング（`anyhow`やカスタムエラー型の利用）への移行を推奨します。

## リファクタリング: UIレイヤーでの冗長な `I18nOps::get().*.clone()` チェーン
`crates/katana-ui/src` 以下のUIコンポーネントにおいて、`crate::i18n::I18nOps::get().settings.title.clone()` のような冗長なメソッドチェーンと `clone()` 呼び出しが100箇所以上散見されます。
これは可読性を下げるだけでなく、微小とはいえ毎フレームのメモリアロケーションを発生させる可能性があります。
翻訳文字列への参照（`&str`）を返すよりエルゴノミックなマクロやヘルパー関数を導入するか、`Rc<String>` / `Arc<String>` 等への移行を検討してください。

## 不具合の可能性: i18nロジックにおけるハードパニック
`crates/katana-ui/src/i18n/logic.rs` において、以下の2箇所でパニックが発生する実装になっています。
- `expect("BUG: Supported language missing from dictionary.")`
- `_ => panic!("BUG: Unhandled language code: {lang}")`

ユーザーの環境や設定ファイル（`settings.json`等）が破損していたり、サポート対象外の言語コードが環境変数から渡されたりした場合、アプリケーション全体が起動時にクラッシュする恐れがあります。
未知の言語コードが渡された場合は、デフォルト言語（例: `en`）にフォールバックするような堅牢な実装への修正を推奨します。
