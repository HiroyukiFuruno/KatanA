# KatanA バージョン管理定義

KatanA プロジェクトのバージョン管理は、セマンティックバージョニング（Semantic Versioning 2.0.0）を基準として運用します。
これは Rust (Cargo) の文化にも合致し、管理を明確にするためのルールです。

## 1. バージョン番号のフォーマット
`v<MAJOR>.<MINOR>.<PATCH>` （例: `v1.2.3`）

## 2. 各バージョンの引き上げ基準
プロジェクト全体のリリースバージョンは以下の基準でアップデートします。

### MAJOR（メジャー）
- **基準**: 後方互換性のない破壊的変更が行われた場合。
  - 例: 設定ファイル（`config.json`等）の構造に互換性のない大きな変更が入った場合、コアAPIの大幅な仕様変更など。
- **備考**: `v0.x.x` の間は初期開発フェーズとし、破壊的変更があっても `MAJOR` は `0` のままとする。`v1.0.0` を「最初の安定版リリース」と定義する。

### MINOR（マイナー）
- **基準**: 後方互換性を保ったまま、新しい機能（Feature）を追加した場合。
  - 例: 「タブ復元機能の追加」「テーマ設定UIの追加」「エクスポート機能の追加」など、ユーザーにとって目に見える新しい価値が提供された場合。
- 複数の一連のタスク（Epic等）を完遂し、新しい機能群としてまとまったタイミング。

### PATCH（パッチ）
- **基準**: 後方互換性を保ったまま、バグ修正や内部リファクタリング、最適化のみを行った場合。
  - 例: 「特定の条件下でのクラッシュ修正」「パフォーマンスチューニング」「CIの修正」など、機能追加を伴わない変更。
- 緊急の不具合や脆弱性対応を含みます。

## 3. OpenSpec（タスク管理）の実践方針
- 大きな機能追加や改善テーマ（Epic的なまとまり）は、原則として **MINOR** バージョンのアップデート（例: `v0.1.0` -> `v0.2.0`）としてスコープを切り出し、管理します。
- 開発途中で生じた細かなバグ等の改修については、**PATCH** バージョン（例: `v0.1.1`）として独立した修正・リリースサイクルを回すことを推奨します。

## 4. リリースライフサイクル
1. 対象バージョンの DoD（Definition of Done）をすべてクリアする。
2. `main` ブランチにすべての変更を統合する。
3. `Cargo.toml` やマニフェストファイルのバージョン番号を更新し、PRをマージする。
4. 対応する Git タグ（例: `v0.1.0`）を付与し、リリースノートを自動/手動で生成する。
5. （OpenSpec管理の場合）当該バージョンのSpecを `archive` ディレクトリへ移行（`.gitignore` の解除による完全解放）する。
