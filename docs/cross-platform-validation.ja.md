# macOSのみ保有している前提での Windows / Linux アプリ検証方針

AIエージェント向け実行ドキュメント

## 目的

macOS 環境しか持たない開発者が、Windows / Linux 向けアプリケーションを現実的かつ継続的に検証できる体制を構築する。

本ドキュメントは、AIエージェントが以下を迷わず判断・実行できるようにするための運用指針である。

- どの検証をどの環境で実施するべきか
- 仮想化、CI、実機確認の役割分担
- egui / eframe ベースのデスクトップアプリにおける注意点
- 実装・検証タスクの優先順位
- リリース前チェック項目

---

## 結論

最適解は以下の三段構えである。

1. **日常検証は macOS 上の仮想環境で行う**
2. **継続的なクロスプラットフォーム検証は CI で自動化する**
3. **最終確認は実機またはクラウド VM で補完する**

この方針により、macOS しか持っていなくても、Windows / Linux 向けアプリの品質を十分に担保できる。

---

## 適用対象

本ドキュメントは、特に以下のようなアプリケーションを対象とする。

- Rust 製デスクトップアプリ
- `egui` / `eframe` ベースの GUI アプリ
- Markdown Viewer / Editor
- Mermaid / PlantUML / Draw.io 等の外部コマンドや描画連携を含むアプリ
- 将来的に macOS / Windows / Linux の3OS対応を前提とするプロダクト

---

## 前提条件

- 開発端末は macOS のみ
- Apple Silicon Mac を想定
- 主開発言語は Rust
- GUI フレームワークは `egui` / `eframe` を想定
- GitHub Actions 等の CI が利用可能
- Windows / Linux 実機は常時保有していない

---

# 検証戦略

## 1. 日常検証は仮想環境で行う

### 方針

手元で頻繁に触る必要がある GUI・基本動作確認は、仮想環境でカバーする。

### 推奨構成

- **Windows**
  - Parallels Desktop 上の Windows 11 ARM
- **Linux**
  - Parallels または UTM / VMware Fusion 上の Ubuntu 系ディストリビューション

### 仮想環境で確認すべき内容

- アプリ起動可否
- 初回表示
- 基本的な画面遷移
- ファイルを開く操作
- Markdown 表示
- スクロール挙動
- 設定画面の基本操作
- 外部コマンド呼び出し
  - `mmdc`
  - `plantuml.jar`
  - その他の CLI 連携
- クラッシュ有無
- ログ出力有無

### 仮想環境での注意点

Apple Silicon 上の Windows は **ARM版** である。
そのため、以下は x86 実機と完全一致しない可能性がある。

- インストーラ挙動
- ネイティブモジュール
- ドライバ依存処理
- 低レイヤー API 呼び出し
- パフォーマンス特性
- 一部の OS 統合機能

### AIエージェントへの指示

- 日常検証の提案では、まず仮想環境で再現可能かを判断すること
- x86 実機との差異が疑われる場合は、仮想環境のみで完結と判断しないこと
- ARM 上で動作した事実をもって「Windows 実機でも問題なし」と断定しないこと

---

## 2. クロスプラットフォームの継続検証は CI で行う

### 方針

OS差異によるビルドエラーやテスト失敗は、人手確認ではなく CI で早期検知する。

### 推奨OSマトリクス

- `macos-latest`
- `ubuntu-latest`
- `windows-latest`

### CI で最低限実施すべき内容

- format check
- lint
- build
- unit test
- integration test
- feature flag ごとの差分確認
- OS別コンパイル可否確認

### Rust の GitHub Actions 例

```yaml
name: cross-platform-check

on:
  push:
  pull_request:

jobs:
  test:
    strategy:
      fail-fast: false
      matrix:
        os: [macos-latest, ubuntu-latest, windows-latest]

    runs-on: ${{ matrix.os }}

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install Rust
        uses: dtolnay/rust-toolchain@stable

      - name: Cache cargo
        uses: Swatinem/rust-cache@v2

      - name: Format check
        run: cargo fmt --all --check

      - name: Clippy
        run: cargo clippy --workspace --all-targets --all-features -- -D warnings

      - name: Build
        run: cargo build --workspace --release

      - name: Test
        run: cargo test --workspace --all-features
```

### CI で確認しやすい論点

- Windows のパス区切り差異
- Linux のファイル権限差異
- OS依存コンパイルエラー
- feature flag の抜け漏れ
- 依存ライブラリ差異
- テストコードが macOS 前提になっていないか

### AIエージェントへの指示

- 「Windows / Linux でも動くはず」という推測ではなく、CI の結果を根拠に扱うこと
- OS差異に関する改修提案では、まず CI 上の再現性を優先して確認すること
- GUI の人手操作が必要な検証と、CI で十分な検証を混同しないこと

---

## 3. 最終確認は実機またはクラウド VM で行う

### 方針

仮想環境と CI だけでは取り切れない、実運用に近い体験差や OS 統合差分を確認する。

### 適した確認手段

- 一時的に借りる Windows 実機
- 小型の Windows PC
- クラウド VM + リモートデスクトップ
- Linux VM / VPS + GUI 環境

### 実機またはクラウドで確認すべき内容

- インストーラ動作
- 日本語 IME
- フォント差異
- DPI / スケーリング
- 右クリックメニュー
- クリップボード
- ドラッグ & ドロップ
- ウィンドウの最大化 / 最小化 / 復元
- OS ネイティブ挙動との差異
- 配布パッケージの実行性
- セキュリティ警告の有無
- GPU 使用時の挙動差

### AIエージェントへの指示

- リリース前の最終品質判断では、実機確認が未実施ならその旨を明記すること
- IME や DPI 問題は CI だけで「問題なし」と判断しないこと
- パッケージ配布前には、少なくとも 1 回は実機またはクラウドで手動確認タスクを提案すること

---

# egui / eframe 前提の補足

## egui の特性

`egui` はクロスプラットフォーム対応しやすいが、OS ネイティブ部品をそのまま利用する設計ではない。
そのため、以下の特徴がある。

### 利点

- 単一コードベースで複数OSに展開しやすい
- Rust だけで閉じた実装を組みやすい
- 描画の一貫性を取りやすい
- 独自 UI を構築しやすい

### 注意点

- OSごとのネイティブ感は薄くなりやすい
- 日本語 IME は重点確認対象
- フォント描画差異が出やすい
- クリップボードや DnD など周辺統合は個別確認が必要
- Wayland / X11 差分など Linux 側の癖が出る場合がある

## AIエージェントへの指示

- egui のクロスプラットフォーム性を過大評価しないこと
- 「ビルドできる」と「違和感なく使える」を分離して評価すること
- Linux デスクトップ差異や IME は、問題発生率が高い領域として重点扱いすること

# タスク分解方針

## 優先順位

### P0: すぐやるべき

- GitHub Actions に Windows / Linux / macOS の matrix を追加
- `cargo build` / `cargo test` / `clippy` / `fmt` を自動化
- Windows 11 ARM 仮想環境を用意
- Linux 仮想環境を用意

### P1: 初期安定化

- OS別の起動確認手順をドキュメント化
- 外部コマンド連携確認
- パス区切り・ファイル権限差異の検証
- フォント差異の把握
- 既知の OS 依存問題を issue 化

### P2: リリース準備

- 配布形式ごとの確認
- IME の重点確認
- DPI / スケーリング確認
- クラウド VM または実機での最終動作確認
- リリース判定チェックリスト整備

# AIエージェントの判断ルール

## ルール1: 推測で「対応済み」と言わない

以下のいずれかが揃っていない場合、対応済みと断定しない。

- 該当OSの CI 成功
- 仮想環境または実機での起動確認
- 必要に応じた手動確認結果

## ルール2: ARM と x86 を区別する

Windows 11 ARM 上で成功しても、x86 Windows 実機と同一とはみなさない。
必要に応じて追加確認タスクを提案する。

## ルール3: egui の「動く」と「違和感なく使える」を分ける

以下を別軸で評価する。

- build できる
- 起動できる
- 操作できる
- 実用上問題ない
- 配布品質に達している

## ルール4: IME・DPI・フォントは重点項目

GUI アプリにおいて、これらは軽視しない。
未確認なら必ず明記する。

## ルール5: 最終的な品質判定では未確認事項を残さない

未検証の論点がある場合は、曖昧な合格判定を出さず、未確認事項として列挙する。

# 推奨運用フロー

## 開発中

1. macOS で開発
2. CI に push
3. Windows / Linux の build / test 結果を確認
4. 必要に応じて仮想環境で GUI を手元確認

## リリース前

1. Windows 仮想環境で手動確認
2. Linux 仮想環境で手動確認
3. 実機またはクラウド VM で最終確認
4. 未確認事項がないことを確認
5. 配布物生成と配布テスト実施

# 参考判断

このテーマにおける実務判断は以下である。

- macOS しかなくても Windows / Linux 検証は可能
- ただし 1 手段だけで完結させない
- 仮想環境は日常確認向け
- CI は継続検証向け
- 実機またはクラウドは最終品質確認向け
- egui はクロスプラットフォームに向くが、OS ネイティブ統合の詰めは別途必要
- IME / フォント / DPI / DnD は重点確認項目

# 最終要約

AIエージェントは、macOSのみの開発環境に対しても悲観する必要はない。
ただし、次の原則を崩してはならない。

- **日常確認は仮想環境**
- **継続品質は CI**
- **最終品質は実機またはクラウド確認**

この三段構えを基本戦略として採用し、未確認事項を曖昧にせず、OS差異を明示しながら品質判断を行うこと。
