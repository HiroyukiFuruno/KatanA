# KatanA

> macOS, Windows, Linux 向けの高速でキーボード駆動型のMarkdownエディタ — Rustとeguiで構築。

KatanAはMarkdownドキュメントの作成とプレビューのためのネイティブデスクトップアプリケーションです。
エディタと併せてコンテンツをリアルタイムにレンダリングし、Diagram-as-Code（Mermaid、PlantUML、Draw.io）をサポート。
ElectronやNode.jsを使わず、単一のネイティブバイナリにより、全体のエクスペリエンスを軽量かつ軽快に保ちます。

---

## 主な機能

- **ライブスプリットビュープレビュー** — 左側で編集し、右側でレンダリングされたHTMLをスクロール同期しながら表示
- **ダイアグラムレンダリング** — Mermaid、PlantUML、Draw.ioのフェンスコードブロックをファーストクラスでサポート
- **GitHub Flavored Markdown** — テーブル、取り消し線、タスクリスト、脚注、自動リンク
- **ワークスペース対応** — フォルダを開き、統合されたファイルツリーからファイルをナビゲーション
- **タブバー** — VSCodeスタイルのタブで複数のドキュメントを同時に開く
- **i18n（国際化）** — UI文字列は完全にローカライズ済み（英語および日本語を同梱）
- **プラグインシステム** — 将来のダイアグラムバックエンド追加のための拡張ポイントを備えたレンダラー基盤
- **AIインテグレーションフック** — 将来的なAIアシストワークフローのためのプロバイダーレジストリ
- **クロスプラットフォームネイティブ** — ネイティブメニューバー(macOS)、アプリ内コマンドUI(Windows/Linux)、CJKフォントサポート

## 開発セットアップ

### 前提条件

- macOS 13 Ventura 以降
- [Homebrew](https://brew.sh) (セットアップスクリプトが見つからない場合はインストールします)

### ワンコマンドセットアップ

プロジェクトのルートから `make init` を実行します。これにより、必要なツールが対話的に確認・インストールされます：

```sh
make init
```

`make init` を公式な入口とし、現状は内部で `scripts/setup.sh` を呼び出します。

スクリプトがインストールするもの：

| ツール | 目的 |
| --- | --- |
| **Homebrew** | パッケージマネージャー (macOS/Linux) |
| **git** (latest) | バージョン管理 |
| **rustup** | Rustツールチェーンマネージャー |
| **Rust stable** + clippy / rustfmt / llvm-tools | コンパイラおよびリンター |
| **cargo-llvm-cov** | ラインカバレッジゲート (CIで100%を強制) |
| **cargo-watch** | ファイル変更時の自動再ビルド |
| **cargo-outdated** | 古い依存関係の検出 |
| **cargo-bloat** | バイナリサイズの分析 |
| **tokei** | ソースコードの行数カウント |
| **lefthook** | Gitフックランナー (pre-commit + pre-push) |

### 主要な `make` コマンド

```sh
make init         # 開発環境を対話的に初期セットアップ
make run          # KatanAのビルドと起動
make test         # フルテストスイートの実行
make check        # 完全なCIチェック: fmt + clippy + テスト
make watch-run    # ファイル変更での自動リロード付きで起動
make doc-open     # APIドキュメントをビルドしてブラウザで開く
```

すべてのリストを確認するには `make help` を実行してください。

### プロジェクト構造

```text
katana/
├── crates/
│   ├── katana-core/      # Markdownパイプライン、ドキュメントモデル、ワークスペース、AIレジストリ
│   ├── katana-ui/        # eguiアプリケーションシェル、プレビューペイン、i18n、スナップショット
│   └── katana-platform/  # ファイルシステム抽象化、設定の永続化
├── scripts/
│   └── setup.sh          # `make init` の実装本体
├── docs/                 # アーキテクチャ決定事項、コーディングルール
├── fixtures/             # テスト用フィクスチャ (サンプルMarkdownファイル)
└── Makefile              # 開発者向けタスクランナー
```

### 品質ゲート

| ゲート | トリガー | チェック内容 |
| --- | --- | --- |
| pre-commit | すべての `git commit` | `cargo fmt --check`, `cargo clippy -D warnings` |
| pre-push | すべての `git push` | フルテストスイート + `cargo llvm-cov` (ラインカバレッジ100%) |
| CI | 毎回のPR / `master`へのPush | fmt · clippy · テスト · カバレッジ · CodeQL セキュリティスキャン |

## 貢献について

KatanAは初期の積極的な開発段階にあります。すべての貢献を歓迎します。

### 貢献の方法

- **バグレポート** — 問題の説明と再現手順を記載して [issue](../../issues) をオープンしてください
- **機能リクエスト** — コードを書く前にアイデアを議論するためにissueをオープンしてください
- **コードの提供** — リポジトリをフォークし、ブランチ上で変更を実装し、プルリクエストを作成してください
- **ドキュメンテーション** — 修正、明確化、翻訳は常に感謝されます

### プルリクエストのガイドライン

1. リポジトリをフォークし、`master` からブランチを作成します
2. PRを開く前に `make check` が完全に通過することを確認してください
3. 新しい振る舞いに対するテストを記述してください (カバレッジゲート: ラインカバレッジ100%)
4. コミットは焦点が絞られたアトミックなものにし、明確で説明的なメッセージを記述してください
5. アプローチについてフィードバックが欲しい場合は、早期にDraft PRを開いてください

### コードスタイル

プロジェクトはlefthookを通じてフォーマットとリンティングを自動的に厳格に強制します。
主要なルールは以下の通りです：

- **rustfmt** — `max_width = 100`, Unixスタイルの改行（`cargo fmt` で強制）
- **clippy** — すべての警告はエラーとして扱う (`-D warnings`); 関数本体の長さ ≤ 30行; 認知的複雑度（Cognitive complexity） ≤ 10
- **i18n** — ユーザーに見えるすべての文字列は `i18n!()` マクロを経由しなければならない — UIテキストのハードコーディング禁止
- 完全なコーディング標準については `docs/coding-rules.ja.md` または `docs/coding-rules.md` (準備中) を参照してください

## クロスプラットフォーム検証とリリースブロッカー

KatanAは主にmacOS上で開発されているため、WindowsおよびLinuxでの安定性を維持するためには厳格な検証ゲートが必要です。

### 1. CI検証 (自動)

GitHub Actions CI (`.github/workflows/ci.yml`) は、クロスプラットフォームのマトリックス (`macos-latest`, `windows-latest`, `ubuntu-latest`) 上で実行されます。
**リリースブロッカー:** いずれかのCIジョブが失敗した場合、そのリリースは公開してはなりません。すべてのプラットフォームで同等の `make check` に通過することが必須です。

### 2. ランタイム起動時動作確認リスト（手動）

必要に応じてマイナーリリースやメジャーリリースの前には、ビルドされた Windows 版 (`KatanA-windows-x86_64.zip`) と Linux 版 (`KatanA-linux-x86_64.tar.gz`) のバイナリを手動で検証する必要があります。これはVM（Parallels、UTMなど）や物理デバイス経由で行われます。

#### Windows および Linux での必須確認事項 (Required Evidence)

1. **初回起動:** 実行ファイルがクラッシュせずに起動し、デフォルトのインターフェースが表示される。
2. **ワークスペースを開く:** ファイルツリー内の「Open expected workspace」などの機能を通じて、ローカルフォルダが正常に開ける。
3. **Markdownの編集:** 既存の Markdown ファイルを編集し、プレビューがリアルタイムに更新されることを確認する。
4. **設定 / アップデート:** 設定ダイアログおよびアップデートダイアログを呼び出し、OS に適した案内が表示されることを確認する（macOS 特有の挙動が漏れていないこと）。

**リリースブロッカー:** この「ランタイム起動時動作確認リスト」で問題が発見された場合、Windows および Linux 向けのリリースはブロックされます。CI（自動テスト）のみでは、UI やフォントの描画の一貫性を完全に保証することはできないためです。
