# 図描画Rust化スパイク 中間レポート

作成日: 2026-04-26

## 結論

現時点では、Mermaid/PlantUMLともに「Rust製（Rust native）なら本線採用」という判断にはしない。

- Mermaidは `merman` が候補としては残るが、安定版としてそのまま採用するには弱い。
- PlantUMLは `plantuml-little` を採用見送り寄りとする。
- Mermaidは公式Mermaid.jsを使い、既存Draw.ioと同じくブラウザ描画・画像化する方式の方が筋が良い。

## Mermaid: merman

評価: あり寄りだが、安定的とは言い難い。

良い点:

- Rust製でアプリ組み込みしやすい。
- 描画までの速度は速く、プレビュー体験としての反応は良い。
- 基本的な図種は出力でき、調整すれば実用レベルに近づけられそう。

問題点:

- KatanA側でSVG後処理が必要になる図種が多い。
- テーマ指定が全図種に素直に効かない。
- クラス図、シーケンス図、状態遷移図、ガントチャート、円グラフで個別補正が必要。
- KatanA側で補正を積み上げると、ライブラリ更新時の差分追従が重くなる。

現時点の見立て:

- KatanA側でハックするか、`merman` をフォーク（fork）して直す必要がある。
- ただし、図種別の補正が増えており、安定版としては微妙。
- Rust製で速いという利点はあるが、「公式Mermaid.jsと同等に崩れない」保証には届いていない。

## PlantUML: plantuml-little

評価: 採用見送り。

確認結果:

- `plantuml-little` のコマンド（CLI）で生成したSVGを、KatanAを通さずブラウザ単体で開いても崩れた。
- そのため、KatanA側のSVG表示処理ではなく、SVG生成側またはテキストレイアウト実装の問題と判断する。

確認した症状:

- 日本語ラベルが極端に短い幅で計算され、文字が重なる。
- 基本的なシーケンス図・アクティビティ図でもノードやラベルが崩れる。
- ダークテーマ以前に、日本語テキストとレイアウトの信頼性が不足している。

SVG内の幅計算例:

- `ユーザー`: `textLength="17.8008"`
- `ファイルを開く`: `textLength="28.9263"`
- `ファイルシステム`: `textLength="35.6016"`

本家Java版PlantUMLでは同じ入力で大きく崩れなかったため、設定不足より `plantuml-little` 側の未成熟を疑うのが自然。

KatanA用途では、Rust製であることより、本家JARと同等に崩れないことを優先する。

## 推奨方針

### Mermaid

公式Mermaid.jsを使う方式を本命候補にする。

既存Draw.ioと同じ考え方で、公式の描画実装をブラウザ上で実行し、結果をキャッシュする。

理由:

- Mermaidの正規実装に追従できる。
- 図種ごとのSVG補正をKatanA側で増やさずに済む。
- テーマ・レイアウト・新図種の追従責務を公式実装側に寄せられる。

懸念:

- ブラウザ描画の起動コストがある。
- オフライン利用や同梱方法を設計する必要がある。
- Draw.io同様、キャッシュと更新戦略が必要。

### PlantUML

本線では本家JAR方式を維持する。

`plantuml-little` は、今回の用途では採用しない。

将来再検討する場合の条件:

- 日本語幅計算が本家PlantUML相当に改善されている。
- 基本フロー図・クラス図・シーケンス図で文字重なりが出ない。
- `skinparam` とダークテーマの反映範囲が実用水準に達している。
- Graphviz前提をKatanA側の配布・設定方針と整合できる。

## 現時点の判断

| 対象 | 判断 | 理由 |
| --- | --- | --- |
| `merman` | 保留。候補には残す | 速いが、KatanA側補正が多すぎる |
| `plantuml-little` | 採用見送り | ブラウザ単体でも崩れ、SVG生成側の問題が強い |
| 公式Mermaid.js | 次の本命候補 | Draw.io方式と同じ責務分離にできる |
| 本家PlantUML JAR | 維持 | KatanA用途では崩れないことが優先 |

## 次の確認項目

- Mermaid.js公式レンダラーをDraw.io方式で呼び出す小さなスパイクを作る。
- キャッシュ単位をソース・テーマ・Mermaid設定で分ける。
- Mermaid.js方式と `merman` 方式の初回描画時間・再描画時間を比較する。
- `merman` は、KatanA側ハックを積む場合とフォーク修正する場合の保守コストを分けて評価する。
