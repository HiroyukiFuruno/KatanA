## 追加要件

### 要件: vendor の独自差分は拡張ブリッジと汎用修正に限定される

KatanA は、`vendor/egui_commonmark_upstream/` に対してローカルで維持する独自差分を、安定した拡張ブリッジと upstream に返せる汎用修正に限定しなければならない。subtree 同期そのものは独自差分ではなく、運用手順として別管理しなければならない。

#### シナリオ: 新しい KatanA プレビュー機能を追加する

- **条件** 保守担当者がタスクリストや検索ハイライトなどの新機能を追加する
- **結果** その機能の実装本体は `crates/katana-ui` 側に追加される
- **結果** vendor 側で許される変更は既存ブリッジ契約への接続または upstream に返せる汎用修正に限られる
- **結果** `katana` 固有命名、製品テーマ、アプリ固有の画面状態は vendor に新規追加されない

### 要件: KatanA 固有の Markdown 振る舞いは `katana-ui` が所有する

KatanA は、テーブル描画、タスクリスト操作、インライン絵文字、リスト項目ハイライト、アンカー収集、検索ハイライト、コードブロック外観といった製品固有の振る舞いを `katana-ui` モジュールとして所有しなければならない。

#### シナリオ: 製品固有の描画差分をレビューする

- **条件** レビュー担当者が Markdown プレビューの差分を確認する
- **結果** 主要な製品固有挙動は `crates/katana-ui/src/markdown_viewer/` 配下で読める
- **結果** プレビューの呼び出し側はファサードを通じて構成される

### 要件: 標準 upstream 描画経路と KatanA 拡張描画経路は切り替え可能である

KatanA は、同一 API 上で標準 upstream 描画経路と KatanA 拡張描画経路を切り替えられなければならない。

#### シナリオ: デグレを切り分ける

- **条件** 保守担当者が描画デグレを調査する
- **結果** 標準描画経路と KatanA 拡張描画経路を比較できる
- **結果** KatanA 拡張描画経路を無効化して upstream の既定挙動を再現できる

### 要件: 描画時の拡張状態は型付きセッション文脈で受け渡される

KatanA は、アンカー収集先、検索状態、アクティブ項目情報などの描画時拡張状態を、追加コールバックや暗黙のグローバル状態ではなく型付きセッション文脈で受け渡さなければならない。

#### シナリオ: アンカー収集と検索状態を同時に扱う

- **条件** 保守担当者がアンカー収集と検索ハイライトの両方に関わる描画変更を行う
- **結果** 必要な状態は単一のセッション文脈から参照できる
- **結果** `pulldown.rs` に機能別コールバックが増殖しない

### 要件: vendor の乖離は運用手順書と監査で管理される

KatanA は、upstream `lampsitter/egui_commonmark` との差分を運用手順書と監査手順で継続管理しなければならない。

#### シナリオ: upstream update を取り込む

- **条件** 保守担当者が subtree update を行う
- **結果** baseline コミット、許可対象ファイル、フィクスチャテスト、差分監査コマンドが運用手順書で確認できる
- **結果** 許可対象外の vendor 変更はレビューでブロックされる

### 要件: 移管した機能はフィクスチャテストで回帰検知できる

KatanA は、ブリッジ経由で移管した機能ごとにテストカバレッジを持ち、プレビュー用フィクスチャで回帰検知できなければならない。

#### シナリオ: 機能移管後の検証

- **条件** テーブル、タスクリスト、絵文字、検索、アンカー、コードブロック外観のいずれかを修正する
- **結果** `katana-ui` 側の単体テストと結合テストで差分が検証される
- **結果** `make check` が品質ゲートとして機能する
