# プロポーザル: UIアーキテクチャの真のコンポーネント化とテスト完全網羅 (UI Architecture & Testability)

## 概要

先行する `v0-8-4` のリファクタリングにおいて、`katana-core` や `katana-platform` といった純粋なロジック層の物理分割・テスト容易性の確保、および全体の AST Linter による行数制限（200行/関数30行）の強制基盤が確立した。しかし、最も巨大であった `katana-ui` レイヤーの分割においては、「ファイル行数をパスさせる」という表層的な対応に留まり、`egui::Ui` を用いた描画処理と純粋なデータ処理（ビジネスロジック）がファイル内で癒着したまま分散するという「偽のコンポーネント化」を引き起こした。

本課題（v0.8.5）では、この UI レイヤーに残された技術的負債を根本から解消する。「egui の画面コンテキストに依存するカバレッジ除外対象の View」と、「テスト可能な純粋な状態管理・計算 Logic」を明確に物理ファイル・モジュールレベルで分断し、真のコンポーネント化を達成する。さらに、分離に成功したロジック層の `COVERAGE_IGNORE` を解除し、カバレッジ100%（UT/IT）を以て「未検証のロジック（バグの死角）」を完全に駆逐する。

## 目標

- `katana-ui` の各巨大コンポーネント（`editor.rs`, `top_bar.rs`, `workspace/panel.rs` 等）を `ui.rs`（描画）と `logic.rs`（状態・計算）等の責務単位へ深く分割し直す。
- `katana-ui/src` 全域に対して、`ast_linter` 制約（ファイル長・関数長・pub_free_fn）を完全に有効化し、`make check` で強制する。
- 分離した純粋ロジックモジュール（`app/`, `state/`, および UI直下の `logic.rs` 等）の `COVERAGE_IGNORE` 指定を解除し、分岐網羅率100%の自動テストを追加する。
- 最終的に `katana-ui` においてもテストによる安全なリファクタリングと品質保証ループが回る「OSSとしての健全性」を確立する。
