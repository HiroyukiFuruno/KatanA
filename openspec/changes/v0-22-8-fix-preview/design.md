## Context

現在、`katana-ui` の Problems パネルにおいて、Linter 等からの Diagnostic に付随する Fix（自動修正）を適用するボタンが存在しますが、これをクリックすると即座にドキュメントにパッチが適用されます。ユーザーは実行前に「何がどのように修正されるのか」を確認できないため、意図しない破壊的変更を防ぐためのセーフティネットが不足しています。LLMを利用したより複雑な自動修正機能を導入する前の DoR（Definition of Ready）として、ユーザーが変更内容を事前にプレビューできる機構が必要です。

## Goals / Non-Goals

**Goals:**
- Problems パネル上の Diagnostic アイテム（Fixが存在する場合）に、修正内容をプレビューできる UI を提供する
- プレビューは軽量かつ直感的な形式（ツールチップまたは展開可能なインライン Diff）で表示する
- 既存の `DiagnosticFix` のデータ構造を利用してプレビュー内容を生成する

**Non-Goals:**
- Fix の適用ロジック自体の変更や最適化
- Problems パネル以外の場所（エディタのメインテキスト領域など）でのインライン Diff プレビュー機能の実装（本タスクでは Problems パネルの拡張に留める）

## Decisions

- **プレビューのUI形式:** 
  - Fixボタンにホバーした際に表示される **Tooltip（ツールチップ）** にプレビューを描画する方式を採用します。
  - 理由: Problems パネルの行の高さを変えずに（レイアウトを崩さずに）、必要なときだけ詳細を確認できるため、最も軽量かつ現在のUIパラダイムに合致するためです。
- **Diff の表現方法:**
  - `DiagnosticFix` は `start_line` から `end_line` までのテキストを `replacement` に置き換える情報を持っています。
  - 削除される元のテキスト（赤色・打消し線）と、追加される新しいテキスト（緑色）を並べて表示するシンプルなインラインテキストの Diff 表現を Tooltip 内に構築します。

## Risks / Trade-offs

- **Risk:** 複数行にわたる巨大な Fix のプレビューが Tooltip 内に収まらない、または見づらくなる可能性がある。
  - **Mitigation:** Tooltip 内のプレビューテキストの行数に上限を設け、長すぎる場合は末尾を省略（`...`）する、または文字サイズを調整するなどのフォーマット制御を行います。
