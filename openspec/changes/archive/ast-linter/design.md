## Context

Katanaプロジェクトでは、100%の単体テスト網羅や独自のコーディング規約（Docs）などの非常に厳しい品質基準が設けられている。しかし、現状「特定のUIメソッドに文字列リテラルを直接ハードコードしてはならない（i18n規約）」「Box<dyn Any>などの型を使ってはならない」といった静的なルールは目視と簡易なgrep検査のみに頼っており、すり抜けが懸念されている。
Rustエコシステムにおいて、通常は `clippy` を拡張するか `dylint` のような動的リンカベースのツールを使用するが、開発環境の維持コストやコンパイル手間の観点から「テストコード内から AST (抽象構文木) を直に解析してルール違反を落とす Linter」を組む手法を採用する。

## Goals / Non-Goals

**Goals:**

- `rustc` の構文解釈と同じ精度（AST・`syn` クレート利用）でソースコードをトラバースし、ハードコード文字列の UI メソッドへのパッシングを検知する。
- ローカルの手元（`cargo test`）と CI ワークフロー、特に `lefthook` による pre-commit/pre-push 時に自動で防波堤として機能させる。
- ルール違反が一つでもあれば、テスト失敗（Exit Code非0）としてコミットをブロックする。

**Non-Goals:**

- マクロ内部の完全な展開と解析（マクロ内まで完璧に追跡するのは `rustc` コンパイラプラグインの領域に踏み込むため、基本は表面上の AST のメソッドシグネチャを検知する）。
- 自動修正機能 (Auto-fix) の提供。開発者が自分で気付いて修正することを優先する。

## Decisions

- **構文解析エンジンに `syn` + `ignore` を採用**
  - *Rationale*: `clippy` の独自ルール追加は Rust コンパイラの内部APIに依存するためメンテコストが高い。一方で `syn` を用いた AST 巡回 (`syn::visit::Visit`) ならば `dev-dependencies` のみで軽量に独自の Linter を `#[test]` として実装できるため。
- **アーキテクチャのプラガブル化**
  - *Rationale*: 当面は i18n のルールが目的だが、将来的に「禁止型の利用」や「30行制限の強制」なども実装できるよう、Visitor に対して複数の Rule チェッカーを走らせる構造とする。
- **Allowlist 機構の実装（ASTレベル）**
  - *Rationale*: UI に描画されるすべての文字列を翻訳対象にすると、区切りタブ `"/"` やアイコン `"🔄"` などまで i18n 辞書に登録する必要が出てきて体験が悪い。AST 上でそれが純粋な記号のみか、数値かなどを判断し、エラーを抑制する設計にする。

## Risks / Trade-offs

- **[Risk] 検知漏れ (偽陰性)**
  - `syn` はあくまで字句から作られた AST を静的解析するため、変数を経由したり、別関数の戻り値として `"Hardcoded"` が渡った場合など、追尾しきれないケースがある。
  - **Mitigation**: 一般的に開発者が直書きしがちな `ui.label("...")` などの「関数の直接引数」をまずは厳密に見張り、抜け穴はコードレビューと併用する。
- **[Risk] パースエラーによるテスト失敗**
  - 未対応の新しい Rust の構文やマクロ記法により `syn::parse_file` がコケる可能性がある。
  - **Mitigation**: `syn` のバージョンを定期的に追従させるとともに、パース失敗時はスキップせずに警告を出力する等でフェイルセーフを設ける。
