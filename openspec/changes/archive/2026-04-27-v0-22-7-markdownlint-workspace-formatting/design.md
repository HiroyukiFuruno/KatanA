## Context

現在の図形描画プレビューは ````` フェンスを直接文字列で見ている。`crates/katana-core/src/preview/section/fence.rs` と `crates/katana-core/src/markdown/fence/mod.rs` の両方にバッククォート前提の処理があり、`~~~mermaid` のようなチルダフェンスは図形として抽出されない。

Lint 設定は `LinterSettings` に `enabled`、`use_workspace_local_config`、`rule_severity` がある。現在の `MarkdownLinterBridgeOps::evaluate_document` は KML の設定ファイル由来の `LintOptions` を読み込んだ後、KatanA 側の `rule_severity` を上書きする。このため「ワークスペース設定を使う」と「KatanA 側で警告/エラー/無視を選ぶ」の責務が混ざり、`.markdownlint.json` の内容が効いているように見えにくい。

エクスプローラーは、ディレクトリとファイルの行にコンテキストメニューがある一方、空き領域には操作入口がない。新規作成の `AppAction::RequestNewFile` / `RequestNewDirectory` は既にあるため、空き領域とヘッダーアイコンは同じ action に接続できる。KML のフォーマット機能はまだアプリ内 action として公開されていないため、ファイル単位・ワークスペース単位の action 境界を追加する。

## Goals / Non-Goals

**Goals:**

- 図形描画は ````` と `~~~` の両方で同じレンダリング結果になる。
- コードブロック生成は、ユーザーがプルダウンでコード種別を選んでから fenced code block を挿入できる。
- Lint の一般設定は、単純なオン/オフと「無視 / 警告 / エラー」の選択に限定する。
- Lint の詳細設定は、通常画面では「ルールの詳細」として扱い、保存先の JSON ファイル名を前面に出さない。
- グローバル設定とワークスペース設定を分け、ワークスペース設定を優先する。
- 一般設定で「無視」にした場合でも、詳細設定の履歴を失わない。
- テーマ、アイコン、Lint のプリセットは、現在値と元にしたプリセットを区別して保存する。
- KML へ渡す設定は、実際に有効な `.markdownlint.json` と KatanA 側の重大度設定を統合した結果になる。
- ファイル単位、ワークスペース単位の Markdown フォーマット操作を UI から実行できる。
- ファイル作成・フォルダ作成は、エクスプローラーのヘッダーと空き領域から起動できる。
- 追加アイコンは各アイコンパック固有の SVG を使う。

**Non-Goals:**

- Lint ルール自体の新規実装。
- `.markdownlint.json` 以外の独自ファイルをワークスペース直下の主要設定保存先にすること。
- フォーマット結果の差分プレビュー機能。
- LLM（local LLM）による自動修正。
- エクスプローラー全体のデザイン刷新。
- テーマやアイコンのマーケットプレイス、外部共有形式、同期機能。

画面上では、通常ユーザーは「Lint を使うか」「どの強さで表示するか」「どのルールセットを使うか」だけを操作する。JSON ファイルは保存形式であり、通常操作の主役にはしない。

## Unified Settings Preset AsIs / ToBe

テーマ（画面の配色）、アイコン（表示する記号セット）、Lint は、現状では UI/UX と保存構造が少しずつ違う。今回分かった課題から見たあるべき姿は、すべて同じ保存仕様で「プリセットを現在設定へ適用する」「変更したらカスタムとして扱う」「元にしたプリセットを表示できる」モデルに統一すること。保存仕様を統一した上で、UI/UX も同じ操作規則へ揃える。

| 対象 | 現状（AsIs） | あるべき状態（ToBe） | 実装で埋める乖離 |
| --- | --- | --- | --- |
| テーマ | `preset`、`custom_color_overrides`、`active_custom_theme` があり、組み込みテーマを少し変えた状態の表示と由来が曖昧になる | 統一保存仕様で、現在テーマ値、元プリセット、変更状態、ユーザープリセットを持つ | テーマの保存構造を統一プリセットモデルへ移行する |
| アイコン | `icon_pack` はテーマ設定側、上書きプリセットはアイコン設定側にあり、アイコンパックと上書きの組み合わせが一つの現在値として分かりにくい | 統一保存仕様で、アイコンパックと個別上書きを一つの現在値として持つ | アイコン設定をテーマ設定から独立させ、統一プリセットモデルへ移行する |
| Lint | プリセット導入時に同じ曖昧さを新しく作る可能性がある | 統一保存仕様で、現在ルール、元プリセット、変更状態、ユーザープリセットを持つ | Lint プリセットを最初から統一プリセットモデルで実装する |
| 既存保存値 | 既に保存済みの値は、元プリセット情報を持っていない | 完全一致する既存プリセットがあればそのプリセット、判断できない場合は「カスタム（元プリセット不明）」として扱う | 移行時に根拠なく元プリセットを推測しない |
| ユーザープリセット | テーマはカスタムテーマ、アイコンは上書きプリセット、Lint は未導入で、対象ごとに意味が違う | すべて「現在設定を名前付きプリセットとして保存し、同種の設定で再利用する」に統一する | テーマ、アイコン、Lint の保存・適用・変更後カスタム表示を揃える |

## Decisions

### 1. 図形フェンスは fence token を抽象化して扱う

````` 専用の処理を、バッククォートとチルダの両方を表す小さな値に置き換える。開始フェンスと終了フェンスは同じ文字種である必要があり、終了フェンスは開始フェンス以上の長さを許容する。これは CommonMark の fenced code block の挙動に合わせる。

コードブロックを新規生成する UI では、自由入力だけにせず、コード種別をプルダウンで選択できるようにする。プルダウンは enum と連動させ、少なくとも `text`、`markdown`、`bash`、`zsh`、`mermaid`、`drawio`、`plantuml`、および開発でよく使う `json`、`yaml`、`toml`、`rust`、`typescript`、`javascript`、`python`、`html`、`css`、`sql` を扱う。表示名と実際に挿入する fence info string は同じ enum から解決し、UI と挿入結果がずれないようにする。

採用理由:

- `~~~` だけを追加する条件分岐より、既存のネスト済みフェンス判定を壊しにくい。
- プレビュー分割と HTML エクスポートの両方へ同じ考え方を適用できる。
- ユーザーがコードブロック種別を手入力で覚える必要を減らし、図形描画ブロックも同じ入口から正しく作れる。

### 2. 一般設定の重大度とルール詳細を分離する

KatanA の一般設定は `RuleSeverity` を使い、表示上の「無視 / 警告 / エラー」を決める。ただし「無視」は KML のルール設定を即時に消す意味ではなく、KatanA 側のワークスペース設定として診断表示から除外する意味にする。

`.markdownlint.json` はルールの有効/無効、詳細値、`default` を持つ。一般設定で「無視」以外へ戻した場合は、保持していた詳細設定を `.markdownlint.json` に戻し、ルール適用を `false` のまま放置しない。

採用理由:

- ユーザーが「警告/エラーの見え方」を変える操作と、`.markdownlint.json` の実体を書き換える操作を分けられる。
- 高度な設定を一時的に無効化しても、再度有効化した時に履歴を復元できる。
- UI 上では JSON を操作しているように見せず、ユーザーの操作対象を「診断の表示」と「ルールの有効/無効」に限定できる。

### 3. 設定の優先順位は workspace > global > default とする

グローバル設定は `~/Library/Application Support/KatanA/.markdownlint.json` などの OS 設定領域に置く。ワークスペース設定はワークスペース直下の `.markdownlint.json` または `.markdownlint.jsonc` を優先する。

通常 UI では JSON ファイルを意識させず、KatanA が管理する共通ルールとして扱う。詳しいユーザー向けには、現在の共通ルールまたは有効ルールをワークスペースへ展開する操作を用意し、展開後はそのワークスペースのルールとして扱う。ワークスペースに既存の `.markdownlint.json` / `.markdownlint.jsonc` がある場合は、それをワークスペースのルールとして利用する。

プリセットは、テーマやアイコンと同じく選べる導線を用意する。ただし Lint では「選択中プリセットへの参照」を持ち続けるのではなく、プリセットを現在のルールへ適用するテンプレートとして扱う。適用後に個別ルールを変更した場合、その設定は現在の共通ルールまたはワークスペースのルールとして保存され、プリセットそのものは変更されない。

組み込みプリセットは次を用意する。

- `KatanA`: おすすめの設定。
- `全て無効`: すべてのルールを無効化する。
- `厳格`: すべての有効ルールをエラーとして扱う。
- `すべて警告`: すべての有効ルールを警告として扱う。

ユーザーは現在のルールをユーザープリセットとして保存できる。ユーザープリセットはグローバルに保存し、別ワークスペースでもテンプレートとして適用できる。ワークスペースに既存のルールファイルがある場合は、それを黙って上書きせず、現在のワークスペースルールとして読み込む。プリセット適用は、既存ワークスペースルールへ反映する明示操作として扱う。

ワークスペース単位の KatanA 重大度設定は、ワークスペースごとに復元できる必要がある。ただし、`.markdownlint.json` は markdownlint 互換の外部設定ファイルであり、KatanA 独自の重大度情報をそのまま混ぜると外部 linter との互換性が崩れる。実装では次のどちらかを選び、選んだ理由をコード近くのテストで固定する。

- `.markdownlint.json` の標準外キーを KML が安全に無視できる場合は、KatanA 用の namespace を設けて同ファイルへ保存する。
- 標準外キーが危険な場合は、既存のワークスペース状態保存先に重大度だけを保持し、`.markdownlint.json` はルール適用・詳細値だけに限定する。

採用理由:

- ユーザー要件の「ワークスペースごとに設定を分ける」と、外部 linter 互換性の両方を守るため。
- KML 側の config 読み込み仕様に依存しすぎない。
- 通常ユーザーには単純な設定 UI を保ちつつ、詳しいユーザーはワークスペース内の標準 markdownlint 設定を直接管理できる。
- テンプレートとしてのプリセットと、保存済みの現在ルールを分けることで、「プリセットを少し変えた状態」が曖昧に残り続けるのを防げる。

### 4. テーマ、アイコン、Lint のプリセット保存仕様と操作を統一する

テーマ、アイコン、Lint は、現在の保存構造が違っていても、ToBe では同じ保存仕様と同じプリセット操作へ統一する。プリセットは直接編集対象ではなく、現在設定へ適用するテンプレートとして扱う。適用後にユーザーが変更した場合は、現在設定をカスタムとして扱い、元にしたプリセットの ID と表示名を保持する。

保存仕様は対象ごとの差分を値の中に閉じ込め、外側の構造を揃える。

- 現在値: 実際に画面へ反映する設定。テーマなら色、アイコンなら icon pack と上書き、Lint ならルール設定。
- 元プリセット: 現在値を作る時に基にした組み込みプリセットまたはユーザープリセット。
- 変更状態: 元プリセットから変更されているかどうか。
- ユーザープリセット一覧: 現在値を名前付きで保存した再利用可能なテンプレート。

UI は、設定プリセット用の再利用ウィジェット（widget: 再利用できる画面部品）へ落とし込む。このウィジェットは「プリセット一覧」「現在状態の表示」「現在値をプリセットとして保存」「元プリセットへ戻す」「詳細設定へ進む入口」を共通で扱う。テーマ、アイコン、Lint は、それぞれの値の読み書きだけをアダプターとして渡し、画面の操作規則はウィジェット側に集約する。

ユーザー操作は、対象がテーマ、アイコン、Lint のどれであっても次の流れに揃える。

- プリセットを選ぶと、現在設定へコピーされる。
- コピー後に個別変更すると、「カスタム（元プリセットを基に変更）」になる。
- 現在設定は、名前を付けてユーザープリセットとして保存できる。
- ユーザープリセットを選んでもリンクは張らず、現在設定へコピーする。
- 組み込みプリセットはユーザー操作で変更されない。
- 詳細設定を開いて個別変更した場合も、同じカスタム状態として扱う。

既存保存値を移行する時は、既知プリセットと完全一致する場合だけそのプリセットとして扱う。完全一致しない場合は「カスタム（元プリセット不明）」にし、根拠なくベースを推測しない。移行では、テーマの `preset` / `custom_color_overrides` / `active_custom_theme`、`theme.icon_pack`、アイコンの `active_preset` / `active_overrides` / `custom_presets` を fixture で固定し、移行前後で見た目が変わらないことを確認する。

採用理由:

- 名前なしカスタム状態を減らし、ユーザーが後から状態を理解できる。
- テーマ、アイコン、Lint で操作規則を揃えられる。
- 組み込みプリセットを誤って書き換える実装を避けられる。

### 5. KML へ渡す設定は KatanA 側で有効設定へ正規化する

KML API が `--config` 相当のファイルパス入力を提供している場合は、それを使う。API がファイルパスを受け取れない場合は、KatanA 側で `.markdownlint.json` / `.markdownlint.jsonc` を読み込み、KML の `LintOptions` または同等の構造へ変換して渡す。

採用理由:

- CLI では `--config` が使えても、ライブラリ API で同じ入口があるとは限らない。
- 設定が効かない不具合を「読み込み漏れ」か「優先順位の上書き」かに分解してテストできる。

### 6. フォーマット操作は action とサービスで分ける

UI は `FormatMarkdownFile(path)` と `FormatWorkspaceMarkdown(root)` のような action だけを発行する。実際の KML 呼び出し、対象 Markdown ファイル列挙、保存、診断再実行、エラー表示は専用サービスに閉じる。

採用理由:

- ファイル行、空き領域、将来のショートカットから同じ処理を呼べる。
- ワークスペース一括フォーマットで UI とファイル走査が密結合しない。

### 7. エクスプローラーの追加アイコンは icon pack ごとに取得する

画像の見た目は「紙アイコンに小さなプラス」「フォルダアイコンに小さなプラス」である。実装では `katana-icon-management` に従い、`scripts/download_icon.sh` で各アイコンパック固有の `file-plus` / `folder-plus` 相当を取得し、`Icon::FilePlus` / `Icon::FolderPlus` として登録する。既存 SVG のコピーで代用しない。

採用理由:

- アイコンパックごとの見た目の一貫性を守れる。
- KatanA 固有アイコンだけに偏らない。

## UI Notes

画面上では、エクスプローラーのフィルターアイコンの左に、小さなファイル追加アイコンとフォルダ追加アイコンが並ぶ。クリックすると、現在のワークスペース直下を親としてファイル作成/フォルダ作成の入力画面が開く。

ファイル行を右クリックした時は、対象が有効な Markdown ファイルなら「ファイルをフォーマットする」が表示される。エクスプローラーのファイルやフォルダがない空白部分を右クリックした時は、「ワークスペース内の Markdown を一括フォーマット」「ファイルの新規作成」「フォルダの新規作成」が表示される。

コードブロックを作る操作では、画面上にコード種別のプルダウンが表示される。ユーザーは `mermaid` や `bash` などを選び、選んだ種別に応じた fenced code block が挿入される。

Lint 設定画面では、通常画面に `.markdownlint.json` / `.markdownlint.jsonc` のファイル名を前面表示しない。ユーザーには「このワークスペースのルール」「共通ルール」「ルールの詳細」として見せる。内部では選択中のルール設定ファイルを読み書きするが、ファイル名やパスは詳細確認、ワークスペースへの展開、エラー時など、必要な場面に限定して表示する。

「このワークスペースのルールを使用する」を切り替えても、画面は一般設定のまま留まる。高度な設定はユーザーが明示的に開いた時だけ表示される。

高度な設定の導線は、既存のアイコン設定と同じ操作パターンを維持する。通常画面から明示的に高度な設定へ入り、全高パネルで検索、展開/折りたたみ、閉じる操作を提供する。ただし意味までアイコン設定と完全に揃える必要はない。アイコン設定の高度な設定は見た目の個別上書きであり、Lint の高度な設定はルール詳細を編集する画面であるため、UI の統一感は操作パターンで担保し、説明文とラベルは Lint の概念に合わせる。

テーマ、アイコン、Lint のプリセット表示は、変更後に「カスタム」と分かる表示にする。元プリセットが分かる場合は「カスタム（KatanA を基に変更）」のように見せる。元プリセットが分からない既存設定は「カスタム」として扱い、根拠のないプリセット名を表示しない。画面構造も、可能な限り「プリセット一覧」「現在の状態」「保存」「詳細設定」の並びに揃える。

## Risks / Trade-offs

- **[Risk] `.markdownlint.json` に KatanA 独自設定を混ぜると、外部 linter が拒否する可能性がある**
  - Mitigation: KML と markdownlint 互換性をテストし、危険な場合は重大度だけ既存 workspace state に保存する。
- **[Risk] 一般設定の「無視」とルール自体の無効化が二重管理に見える**
  - Mitigation: UI 文言で「診断で無視」と「ルール自体を無効」を分け、設定ファイル名ではなく操作結果で説明する。
- **[Risk] ワークスペース一括フォーマットが大きな差分を生む**
  - Mitigation: 対象は Markdown ファイルに限定し、実行後に変更ファイル数と失敗ファイルをステータス表示する。
- **[Risk] `~~~` 対応で既存のネスト済みコードブロック回避が壊れる**
  - Mitigation: バッククォートとチルダ双方で、ネスト済み diagram が誤抽出されない回帰テストを追加する。

## Migration Plan

1. `~~~mermaid`、`~~~plantuml`、`~~~drawio` の失敗テストを追加する。
2. フェンス抽出をバッククォート/チルダ両対応にする。
3. Lint 設定の責務を一般設定、詳細設定、グローバル設定、ワークスペース設定へ分ける。
4. KML に渡す effective config をテストで固定する。
5. ファイル単位のフォーマット action とサービスを追加する。
6. ワークスペース一括フォーマット action とサービスを追加する。
7. エクスプローラーのファイル行と空き領域へコンテキストメニューを追加する。
8. エクスプローラーヘッダーへファイル追加/フォルダ追加アイコンを追加する。
9. 追加アイコンを全アイコンパックに登録する。
10. UI スナップショットと `make check` で検証する。
