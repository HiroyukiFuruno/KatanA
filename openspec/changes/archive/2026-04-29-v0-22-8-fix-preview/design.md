## Context

現在、`katana-ui` の Problems パネルでは、Diagnostic に付随する Fix（自動修正）を押すと即座にドキュメントへパッチが適用されます。単体の修正であれば軽量なホバー表示でも補助できますが、ファイル単位・複数ファイル・ワークスペース全体の一括修正では、適用前に git diff 風の差分画面を挟み、ユーザーがファイルごとに取り込むか拒否できる必要があります。

この差分確認は、手動の一括修正と今後の LLM 自動修正の共通の受け皿にします。LLM が生成した修正でも、ユーザーは「ファイルに何が入るか」を見てから反映できる状態にします。

## Goals / Non-Goals

**Goals:**

- ファイル単位の修正適用前に、git diff 風の差分確認画面を表示する
- 差分確認画面では Split（左右分割）と Inline（行内表示）を切り替えられるようにする
- 初期表示は Split とし、設定で永続的に既定値を変更できるようにする
- 差分画面内の切り替えボタンは、その画面だけの一時切り替えとして扱う
- 単一ファイル、複数ファイル、ワークスペース全体の修正を同じ差分確認フローで扱う
- 複数ファイルの場合は、ページ送りのようにファイルごとの差分を順に表示し、反映または拒否を選べるようにする
- Problems パネルの単体 DiagnosticFix ホバープレビューは、補助機能として残す

**Non-Goals:**

- LLM の修正生成ロジックそのもの
- git のパッチ適用エンジンを直接使った高度な3-way merge
- 差分画面内での部分行・部分ハンク単位の採用

## Decisions

- **主UI形式:**
  - ファイル単位の差分確認画面は、再利用可能な `DiffViewer` コンポーネントとして実装します。
  - 初期の表示先はモーダルですが、将来のタブ表示や LLM チャット内表示でも同じビューアを使える責務分離にします。
  - 右下に「キャンセル」と「修正を反映」を配置します。
  - 複数ファイルの場合は、現在のファイル番号と総数を表示し、「前へ」「次へ」で移動できるようにします。
- **差分ビューア:**
  - 既存の分割表示に近い「左コード / 右コード」の表示面にします。
  - 左右それぞれに行番号を表示し、修正箇所の行数を追えるようにします。
  - ファイル見出しには追加行数・削除行数を表示します。
  - 連続する未変更行は折りたたみ表示し、展開操作の受け皿を持たせます。
  - UI/UX が劣化しない操作はアイコンボタンにします。
- **差分データ:**
  - 修正適用前の本文と、DiagnosticFix 群または将来の LLM 修正が生成した適用後本文を比較し、表示用の Diff モデルを作ります。
  - 既存の `ApplyLintFixesForFiles` は即時適用せず、差分確認用の pending state（保留状態）に変換します。
- **Split 表示:**
  - 左に修正前、右に修正後を表示します。
  - 削除行は左側、追加行は右側で強調します。
- **Inline 表示:**
  - git diff に近い1列表示とし、削除行と追加行を上下に並べます。
- **表示方式の設定:**
  - 永続設定は `DiffViewMode` として保存します。
  - 初期値は Split です。
  - 差分画面内の切り替えは、保存ボタンを伴わない一時切り替えです。
- **補助UI:**
  - Problems パネルの単体「修正」ボタンに対するホバープレビューは残します。
  - ただし、クリック時の正式な適用前確認はファイル差分画面に統一します。
- **KMLとの責務境界:**
  - KML 側へ相談する場合は、KatanA 固有の差分画面や承認 UI ではなく、汎用的な Fix 適用 API に限定します。
  - KatanA は差分画面、ページ送り、承認、設定を引き続き所有します。
  - KML 側で採用されない可能性があるため、本変更は KatanA 側だけでも完結できる構成にします。
  - 将来の重複削減候補として、KML issue #43（汎用 Fix 適用プレビュー API）を作成済みです。

## Risks / Trade-offs

- **Risk:** 巨大なファイル差分で描画負荷が高くなる。
  - **Mitigation:** 差分モデルをファイル単位で構築し、複数ファイルでは現在表示中のファイルだけを描画対象にします。
- **Risk:** 一括修正で一部ファイルだけ拒否した時に、適用済み・未適用の状態が分かりづらくなる。
  - **Mitigation:** ファイルごとに「未判断」「反映済み」「拒否済み」を明示し、最後に残件を表示します。
- **Risk:** 既存の即時修正に慣れたユーザーには1ステップ増える。
  - **Mitigation:** 破壊的変更を避けるため、ファイル単位以上の修正は確認画面を必須とします。単体ホバーは補助確認として残します。
