## ADDED Requirements

### Requirement: オーナー限定のリリースパイプライン実行権限
OSS リポジトリとして公開されるにあたり、CI（テスト等）と CD（DMG生成および GitHub Releases作成）のワークフローは明確に分離されなければならない（MUST）。CD 処理はオーナーユーザーのみが手動で発火、もしくは承認プロセスを経てのみ実行されなければならない（MUST）。既存の CI ワークフロー（`.github/workflows/ci.yml`）はそのまま維持し、リリース用の CD ワークフローを別ファイルとして新設する形をとること。

#### Scenario: 悪意のある/不要なリリース実行の防止
- **WHEN** フォークされたリポジトリからの Pull Request が作成され、または第三者がパイプラインを起動しようとしたとき
- **THEN** 既存の CI（テスト・リント・CodeQL・カバレッジ）のみが走り、CD ワークフロー（GitHub リリース作成・DMGアップロード等）は `workflow_dispatch` または `github.actor` チェックにより実行がブロック・スキップされる。

### Requirement: インストール前提のドキュメント再構築
ルートディレクトリにある `README.md` と `README.ja.md` は、一般ユーザーが DMG から直接インストールすることを前提とした内容に書き直されなければならない（MUST）。既存の開発者向けドキュメント（`README.md` の開発ガイド内容 + `docs/coding-rules.ja.md` 等）は `docs/` ディレクトリに保持・集約されなければならない（MUST）。

README には以下の要素が含まれなければならない（MUST）:
- アプリケーションアイコン画像の表示（`assets/icon.iconset/` 内の適切なサイズ画像等）
- 現在のバージョン番号の明示
- Shields.io によるバッジ群（ライセンス、CIビルドステータス、最新バージョンタグ、プラットフォーム対応状況など）

#### Scenario: ユーザーによる README の参照
- **WHEN** リポジトリを訪れた一般ユーザーが README を読んだとき
- **THEN** アプリアイコン、シールドバッジ（ライセンス、ビルド状態、バージョン等）、機能の紹介、最新の DMG のダウンロードリンク先（Releases）、ならびにインストール方法と未署名アプリの起動方法が明確に案内され、同時に開発者向けドキュメント（`docs/` 配下）へのリンクが提供される。

### Requirement: リポジトリの OSS 化 (Public 化)
リリースエコシステムとドキュメント一式（ユーザー向けの `.dmg` インストールフローおよびコントリビュート案内）が完備された段階で、当該ソースコードリポジトリは全世界に対して公開ステータスへ移行されなければならない（MUST）。

#### Scenario: 最終リリースの前の OSS 化の実施確認
- **WHEN** 全機能タスクが完了し、最終チェックフェーズ（DoD の確認前または運用直前）に至ったとき
- **THEN** リポジトリの設定画面から「Change visibility」を Private から Public へ変更し、最終の GitHub Releases 作業がパブリックな環境上で無事成功するかを検証する。
